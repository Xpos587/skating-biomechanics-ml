# S3-Only Storage Migration Design

**Date:** 2026-04-11
**Status:** Approved

---

## Goal

Eliminate `outputs_dir` (local `data/uploads/`) entirely. All files stored in S3-compatible object storage (Cloudflare R2 now, self-hosted MinIO later). Backend acts as streaming proxy for file serving, abstracting the storage backend from the frontend.

## Motivation

- **Single storage backend** — no dual-path (local + R2) logic
- **Serverless-ready** — backend doesn't need local disk for processing results
- **Portable** — switching R2 → MinIO requires only config change
- **Resilient** — backend proxy works even if Cloudflare is blocked in Russia (frontend never talks to R2 directly)

## Architecture

```
Current:
  Frontend → POST /detect → save to outputs_dir → RTMPose → return local path
  Frontend → POST /process (SSE) → local GPU → outputs_dir → relative path
  Frontend → POST /process/queue → arq → Vast.ai ↔ R2 → download to outputs_dir
  Frontend → GET /outputs/{file} → FileResponse from disk

New:
  Frontend → POST /detect → upload to R2 (input/) + RTMPose on /tmp → return R2 key
  Frontend → POST /process/queue → arq → Vast.ai ↔ R2 (no local files)
  Frontend → GET /outputs/{key} → StreamingResponse from R2
  SSE endpoint: REMOVED
```

## R2 Key Structure

```
input/{uuid}.mp4                    # Uploaded videos (from detect endpoint)
output/{uuid}_analyzed.mp4          # Result videos (from Vast.ai worker)
output/{uuid}_poses.npy             # Pose exports
output/{uuid}_biomechanics.csv      # CSV exports
```

UUID is generated by the detect endpoint and used throughout the pipeline.

## File Serving Strategy

**Backend streaming proxy** (not presigned URLs):

1. Frontend requests `GET /api/v1/outputs/{key}`
2. Backend calls `storage.stream_object(key)` → boto3 streaming body
3. Backend returns `StreamingResponse` with chunked transfer
4. No buffering, no CORS on bucket, no frontend knowledge of R2

Why not presigned URLs:
- Cloudflare may be blocked in Russia — proxy avoids this
- Future MinIO migration — only config change, zero frontend changes
- No CORS configuration needed on the bucket

## Changes by Component

### 1. `src/storage.py` — Expanded storage layer

Add:
- `upload_bytes(data: bytes, key: str) -> str` — upload from memory (for detect endpoint streaming)
- `stream_object(key: str) -> tuple[StreamingBody, int, str]` — return (body, content_length, content_type) for proxy
- `get_object_url(key: str, expires: int = 3600) -> str` — presigned URL for future direct access
- `list_objects(prefix: str) -> list[str]` — list keys with prefix (for cleanup)
- `object_exists(key: str) -> bool` — check if key exists

Keep existing: `upload_file()`, `download_file()`, `delete_object()`, `_client()`

### 2. `src/config.py` — Remove outputs_dir

- Remove `outputs_dir` from `AppConfig`
- Add `presign_expires: int = 3600` to `R2Config`
- Remove from `.env.example` and `.envrc`

### 3. `src/backend/routes/detect.py` — Upload to R2

Current: saves to `outputs_dir/detect_XXXXXX.mp4`, returns `video_path` (absolute local path).

New:
1. Save uploaded file to `tempfile.NamedTemporaryFile()`
2. Upload to R2 `input/{uuid}.mp4` using `storage.upload_bytes()`
3. Run RTMPose on `/tmp` file (CPU is fine for 30-frame preview)
4. Return `video_key: str` instead of `video_path: str`
5. Cleanup `/tmp` file in `finally` block

### 4. `src/backend/routes/process.py` — Remove SSE, queue only

Remove:
- `POST /process` (SSE endpoint) — entire function, `_cancel_event`, `_executor`, `OUTPUTS_DIR`
- `PipelineCancelled` import and handling
- `PersonClick` import
- `process_video_pipeline` import
- `ThreadPoolExecutor` import

Modify `POST /process/queue`:
- Accept `video_key: str` instead of `video_path: str` in `ProcessRequest`
- Pass `video_key` to arq job

Keep unchanged:
- `GET /process/{task_id}/status` — polling endpoint
- `POST /process/{task_id}/cancel` — cancel endpoint

### 5. `src/backend/routes/misc.py` — Proxy from R2

Current: `FileResponse(str(file_path))` from `OUTPUTS_DIR / filename`.

New:
- `GET /outputs/{key:path}` calls `storage.stream_object(key)`
- Returns `StreamingResponse(iterable=body.iter_chunks(), content_type=..., headers={"Content-Length": ...})`
- Returns 404 if key doesn't exist (`storage.object_exists()` or catch botocore exception)
- Remove `OUTPUTS_DIR` module variable

### 6. `src/worker.py` — Thin dispatcher, no local GPU

Remove:
- Local GPU fallback (`process_video_pipeline` call, lines 160-180)
- `outputs_dir` creation and usage (lines 106-108)
- Relative path calculations (lines 188-204)

Modify `process_video_task()`:
- Accept `video_key: str` instead of `video_path: str`
- Call `process_video_remote(video_key=video_key, ...)` — R2 key, not local path
- Store R2 keys directly in Valkey result (no local path resolution)

### 7. `src/vastai/client.py` — Simplify (no upload, no download)

Current: uploads video to R2, routes to Vast.ai, downloads results to local.

New `process_video_remote()`:
- `video_key: str` parameter (R2 key, not local path)
- Skip upload step — video already in R2 from detect endpoint
- Pass `video_r2_key: video_key` directly to Vast.ai worker
- Skip download step — results stay in R2
- Return `VastResult` with R2 keys instead of local paths:
  ```python
  @dataclass
  class VastResult:
      video_key: str       # was video_path
      poses_key: str | None  # was poses_path
      csv_key: str | None    # was csv_path
      stats: dict
  ```

### 8. `src/backend/schemas.py` — Update request schema

- `ProcessRequest.video_path` → `ProcessRequest.video_key`
- `DetectResponse.video_path` → `DetectResponse.video_key`
- `ProcessResponse.video_path` → `ProcessResponse.video_path` (keep as-is — it's now a relative R2 key used by `/outputs/` endpoint)

Actually: keep `video_path`, `poses_path`, `csv_path` field names in `ProcessResponse` — they now hold R2 keys instead of relative local paths. The frontend uses them to construct `/api/v1/outputs/{video_path}` URLs, which still works because the backend proxies from R2. This minimizes frontend changes.

### 9. Frontend — Minimal changes

The frontend constructs URLs like `/api/v1/outputs/${result.video_path}`. Since the backend now proxies from R2, these URLs still work. The field name `video_path` in the response is now semantically an R2 key, but the URL construction is identical.

Only changes needed:
- `DetectResponse.video_path` → `DetectResponse.video_key` (backend returns R2 key from detect)
- Analyze page: read `video_key` from URL params instead of `video_path`
- Process request: send `video_key` instead of `video_path`

### 10. `vastai/server.py` — No changes

Already R2-native. Receives R2 key, processes, uploads results to R2.

### 11. `src/web_helpers.py` — No changes

`process_video_pipeline()` is only called by `vastai/server.py` inside the container. It uses `tempfile.TemporaryDirectory()` and writes to `output_path` locally inside the container. The container then uploads results to R2.

## Risks & Mitigations

1. **R2 latency for detect** — RTMPose runs locally on /tmp, only upload to R2 adds latency (~2-5s for typical video). Acceptable for detect endpoint.
2. **R2 unavailability** — detect fails if R2 is down. Mitigate: health check on startup.
3. **No local GPU fallback** — if Vast.ai is down, processing fails. Mitigate: arq retry mechanism (30s, 120s delays).
4. **Large video proxy** — backend streams chunks, no buffering. Memory usage is constant (~8KB per chunk).
