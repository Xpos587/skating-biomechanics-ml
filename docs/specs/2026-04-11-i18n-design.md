# Frontend i18n Design

> **Date:** 2026-04-11
> **Status:** Approved

## Goal

Replace all hardcoded Russian strings in the Next.js frontend with i18n using next-intl. Backend remains in Russian (SSE messages, recommendations, error responses).

## Decisions

| Decision | Choice | Why |
|----------|--------|-----|
| Library | next-intl | Native App Router + Server Components + ICU plurals + ~2KB |
| Routing | Cookie-based, no URL prefix | SaaS tool, not public site — SEO irrelevant |
| Default locale | `ru` | Primary audience |
| Supported locales | `ru`, `en` | Initial scope |
| Key format | Flat with dot prefix (`common.signIn`) | Readable, grep-friendly |
| Locale source | `User.language` from DB (auth) → cookie → `Accept-Language` → fallback `ru` | Already have user field |
| Backend i18n | None | SSE messages, recommendations stay Russian |
| Scope | All frontend pages (~81 strings, 15 files) | Complete coverage |

## Architecture

### File Structure

```
src/frontend/
├── messages/
│   ├── ru.json
│   └── en.json
├── i18n/
│   ├── request.ts       # getRequestConfig — reads cookie
│   ├── routing.ts       # defineRouting (no locale prefix)
│   └── navigation.ts    # createNavigation from routing
├── src/i18n/
│   └── index.ts         # getLocale(), setLocale() client helpers
├── middleware.ts         # Thin wrapper: i18n middleware + auth check
└── src/app/
    └── layout.tsx       # NextIntlClientProvider with messages
```

### Locale Resolution Flow

```
Request → middleware.ts
  → Read skating_lang cookie
  → Not found? Read Accept-Language header
  → Not found? Fallback: "ru"
  → Set cookie if not present
  → Continue to page

Login → PATCH /api/v1/users/me/settings (language field)
  → Response sets skating_lang cookie
  → Frontend re-renders with new locale

Language switch in Settings →
  → PATCH /api/v1/users/me/settings { language: "en" }
  → setLocale("en") updates cookie + triggers re-render
```

### next-intl Configuration (no prefix)

```typescript
// i18n/routing.ts
import { defineRouting } from "next-intl/routing"

export const routing = defineRouting({
  locales: ["ru", "en"],
  defaultLocale: "ru",
  localePrefix: "never",   // No /ru/ or /en/ in URLs
})
```

```typescript
// i18n/request.ts
import { getRequestConfig } from "next-intl/server"
import { cookies } from "next/headers"
import { routing } from "./routing"

export default getRequestConfig(async () => {
  const cookieStore = await cookies()
  const locale = cookieStore.get("skating_lang")?.value ?? routing.defaultLocale
  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  }
})
```

### Client Helpers

```typescript
// src/i18n/index.ts
"use client"
import { useLocale, useTranslations as useT } from "next-intl"

export { useLocale }

export function setLocale(locale: string) {
  document.cookie = `skating_lang=${locale};path=/;max-age=31536000;samesite=lax`
  window.location.reload()
}

// Wrapper for consistent namespace usage
export function useTranslations(namespace: string) {
  return useT(namespace)
}
```

### Component Usage

```tsx
// Before
<h1>Вход</h1>
<Button>{loading ? "Вход..." : "Войти"}</Button>

// After
import { useTranslations } from "@/i18n"

export default function LoginPage() {
  const t = useTranslations("auth")
  return (
    <h1>{t("signIn")}</h1>
    <Button>{loading ? t("signingIn") : t("signInBtn")}</Button>
  )
}
```

### Server Component Usage

```tsx
// app/layout.tsx (Server Component)
import { getTranslations } from "next-intl/server"
import { NextIntlClientProvider } from "next-intl"

export default async function RootLayout({ children }) {
  const messages = (await import("../messages/ru.json")).default
  return (
    <NextIntlClientProvider messages={messages}>
      {children}
    </NextIntlClientProvider>
  )
}
```

### Validation Messages (Zod)

```typescript
// Zod errors from backend come in Russian (detail field).
// Frontend validation messages are in i18n:
// lib/auth.ts keeps backend error passthrough,
// client-side validation uses t() from component context.

// Before (in auth.ts):
email: z.string().email("Введите корректный email")

// After: move validation to component, use t():
const schema = z.object({
  email: z.string().email(t("validation.emailInvalid")),
})
```

## Implementation Notes

1. **Zod validation messages** (`lib/auth.ts`): Move client-side validation messages to components where `useTranslations` is available. Backend error messages (from `detail` field) stay as-is — they come from FastAPI in Russian.

2. **Toast messages**: Components call `toast.success(t("auth.signInSuccess"))` — translations evaluated at call time.

3. **Dynamic values**: Use ICU syntax for interpolation: `"Обнаружено людей: {count}"` → `t("home.personsDetected", { count: resp.persons.length })`.

4. **Pluralization**: Not needed for initial scope (no count-based strings currently), but next-intl supports ICU plurals if needed later.

5. **`document.title` / metadata**: Set via `next-intl/server` `getTranslations()` in layout.tsx.

6. **Middleware ordering**: i18n middleware runs first (reads cookie, sets locale), then auth middleware checks protected routes.

## What Stays in Russian

- Backend API error messages (`detail` field in responses)
- SSE event messages (`Анализ завершён!`, `Обработка отменена`)
- Recommender output (biomechanics recommendations in Russian)
- Validation messages from backend (e.g., "Email already registered")
